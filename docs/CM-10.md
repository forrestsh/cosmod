# CM-10: Publish Crystal Garden v0.13 as a Cosmic Grid application

## Summary

As the site manager, I want to publish **Crystal Garden v0.13** — an interactive demonstration of the Cosmic Grid framework — as a discoverable application within the Cosmod site, so that visitors can explore 59 rules across 7 algorithm stages directly from the homepage.

## Background

Crystal Garden is a single-file HTML toy that runs a 40³ face-centered-cubic (FCC) lattice universe in the browser. Starting from v0.3 with 4 rules, **v0.13 has accumulated 59 rules across 7 algorithm stages** — from the simplest Oh-symmetric crystals through a complete 26-letter English alphabet, **and now to the first real 3D scan data (a GSO toy dog mesh)**.

The core thesis comes from the Cosmic Grid framework: **complex 3D morphologies can emerge from extremely simple local rules**. The major breakthrough in v0.13 is that a real-world 3D model — a 536-node toy dog — can be evolved from **1 seed + 38 steps** of local rules to 89.2% recall. This is the first time the tool has left the "geometric primitives" and "letters" domain to directly tackle natural form.

The deliverable file already exists at the repo root:

- `crystal_garden_v13.html` — single file, ~95 KB, depends on Three.js via CDN.

The corresponding write-up is `README_v13.md`.

## User-facing behavior

1. A new card on the homepage links to Crystal Garden v0.13 alongside the existing simulation cards (Hex grid, Toroidal grid, Cosmic grid 4x4, Magic square rule search) and the Book card.
2. Clicking the card opens the Crystal Garden experience in the browser. No backend, no build step — the app runs entirely client-side.
3. **Desktop:** drag to rotate, scroll to zoom. Click rule cards at the bottom to switch rules. `Space` plays/pauses, `N` steps, `R` resets.
4. **Mobile:** one-finger drag to rotate, pinch to zoom. Horizontally scroll the bottom rule strip. Buttons control evolution.
5. The voxel shape can be toggled (top-right) between cube and rhombic dodecahedron — the latter being the natural Voronoi cell of the FCC lattice.
6. All 59 forms are organized across 7 algorithm stages, each browsable from the rule selector:
   1. **Single-seed Oh** (sym144) — 6 forms, convex Oh-symmetric crystals.
   2. **Carving (v3)** — 5 forms, hollow shells / wireframes / stella octangula.
   3. **D4h** (sym354) — 4 forms, cylindrical / prismatic / capsule shapes.
   4. **Multi-seed** (sym144 + N seeds) — 8 forms, cubic cages / sphere arrays / lattice clusters.
   5. **C1 optimized seeds** (sparse 4096) — 6 forms, true asymmetric (L / F / T / dumbbell).
   6. **Alphabet** (sparse 4096) — 26 uppercase letters A–Z.
   7. **Real 3D mesh** (sparse 4096) — 1 form, the GSO toy dog (536 nodes, 89% recall).

## Site integration expectations

1. The HTML file `crystal_garden_v13.html` is served as a static asset, similar to how the books are served at `/books/`. A clean URL path (e.g. `/crystal-garden/` or `/crystal-garden-v13/`) is preferred over a raw filename.
2. Vercel routing must serve the static file directly — the SPA catch-all rewrite must exclude this path (mirroring the `/books/` exclusion in `vercel.json`).
3. The homepage card uses the existing card pattern in `src/HomePage.jsx` (consistent with the other simulation cards and the Book card from CM-9).
4. The card title and description should communicate what the user will see (e.g. "Crystal Garden v0.13" / "59 rules · 7 stages · FCC crystals, alphabet, and a real 3D dog mesh").
5. No content modifications to `crystal_garden_v13.html` itself — it ships as authored.
6. `README_v13.md` is published alongside the app (linked from within the experience or referenced from the homepage card description) so visitors can read the framework context.

## Acceptance criteria

1. A new card titled **Crystal Garden v0.13** appears on the homepage and links to the Crystal Garden experience.
2. The Crystal Garden URL serves `crystal_garden_v13.html` as a static page; the React SPA does not intercept the route.
3. On both desktop and mobile, the experience loads with all 59 rules selectable across 7 stages.
4. The voxel shape toggle (cube / rhombic dodecahedron) is functional.
5. Keyboard controls (`Space` / `N` / `R`) and mouse / touch interactions work as documented.
6. The deployment is reachable on the production Vercel domain without console errors that block rendering.
7. `README_v13.md` is accessible to readers who want the framework context behind the demo.

## Representative content highlights (informational)

These are the headline demonstrations a visitor should be able to find and run:

- **rt_FCC · 3-cycle** — the 3-non-trivial-mapping lookup table that supports a light-speed gun, AND gate, and NOT gate on FCC, establishing Turing completeness.
- **sym144 stage 1–2** — convex Oh shapes (octahedron, cube, cuboctahedron, RD, truncated octahedron) and carved variants (hollow shells, wireframes, stella octangula).
- **D4h stage 3** — cylindrical / prismatic / capsule shapes via symmetry breaking.
- **Multi-seed stage 4** — cubic cage (504 nodes from 8 corner seeds), 6-sphere octahedral array, edge-decorated lattices.
- **C1 + optimized seeds stage 5** — pyramid (98%), L-shape (100%), F (100%), T (100% recall), arrow (95%), dumbbell (100% recall).
- **Alphabet stage 6** — 26 uppercase letters; 15 letters at 100% perfect, 24 at ≥95% recall, average 98.7%.
- **GSO toy dog stage 7** — 1 seed + 38 steps → 478 of 552 nodes belong to the target (89.2% recall) on a 536-node FCC voxelization (d_nn = 4mm).

## Known limitations to call out (informational)

These are the same caveats called out in `README_v13.md` and should not block publishing — but the homepage description should not over-promise:

- All 59 forms are pre-computed rule sequences, **not** real-time search. The browser plays back; it does not search.
- No custom seed placement UI, no custom rule editor, no STL export, no word-spelling combiner.
- The 40³ bounding box (expanded from 30³) accommodates the dog's z range but is fixed.
- Letter S tops out at 89%; the dog tops out at 89.2% recall — this is the inherent ceiling of single-layer C1 greedy rules on dense natural surfaces, not a placement-tuning problem.

## Out of scope (unless amended)

- Custom seed-placement UI, custom rule editor, STL export, word-spelling UI — flagged in the README as v0.14 candidates.
- Modifying `crystal_garden_v13.html` content (rules, UI, or geometry).
- Multi-scale or position-dependent rule algorithms aimed at breaking the 89% ceiling.
- Any server-side or backend integration — the toy is single-file, browser-only.
